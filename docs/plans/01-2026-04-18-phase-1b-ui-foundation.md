# Phase 1B: UI Foundation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Spec:** [`docs/specs/01-monorepo-foundation.md`](../specs/01-monorepo-foundation.md) — Phase 1B only.

**Prerequisite:** Phase 1A plan (`docs/plans/01-2026-04-18-phase-1a-backend-scaffold.md`) merged to `main`. If you have not completed 1A, stop — 1B edits the Rails app scaffold that 1A produces.

**Goal:** Stand up the FaleCom UI foundation on the Rails 8.1.3 app: Vite + Tailwind CSS 4.0.2 replace the asset pipeline, ViewComponent + Jetrockets UI (JR Components) deliver the design system, a base layout with Navbar + Sidebar wraps every view, the Rails 8 authentication login page renders through the JR form builder, and an empty authenticated dashboard page confirms the full auth → layout → Vite HMR loop works.

**Architecture:** Rails renders HTML; Vite (via `vite_rails`) builds and serves JS + CSS with HMR. Tailwind CSS 4.0.2 compiles through the `@tailwindcss/vite` plugin. ViewComponent owns presentational logic; JR Components live under `packages/app/app/components/ui/` — copied into the repo (no gem dependency), owned source. Hotwire's Turbo + Stimulus are the interactivity layer; there is no React / Vue / Svelte anywhere in 1B. The login flow uses Rails 8's built-in authentication (sessions controller + cookie) from 1A, restyled through JR form fields.

**Tech Stack:** Rails 8.1.3 (from 1A), vite_rails ~> 3.0, Tailwind CSS 4.0.2, `@tailwindcss/vite`, view_component ~> 3.x, Stimulus (from `@hotwired/stimulus`), foreman (system gem), JR Components (copied from https://ui.jetrockets.com/ui/getting_started).

**Executing model:** Sonnet 4.6 with low thinking — plan is prescriptive; external steps that rely on the JR guide include explicit "follow their steps exactly, then verify these files exist" so the executor detects divergence.

**Prerequisites to execute:**
- Phase 1A is on `main`; local `main` is up to date.
- Docker Desktop running, workspace + postgres containers up (`docker compose -f infra/docker-compose.yml up -d workspace postgres`).
- Dev user `dev@falecom.test` / `falecom-dev-password` exists (Phase 1A's `bin/setup` created it; re-run `bin/setup` if uncertain).
- Network available from inside the workspace container (npm + bundler need to fetch).

**Conventions inside this plan:** same as Phase 1A — `Run (host):` and `Run (workspace):` prefixes, `docker compose -f infra/docker-compose.yml exec workspace bash -lc '<cmd>'` for the workspace variant, absolute paths under `/Users/cooper/dev/falecom`.
